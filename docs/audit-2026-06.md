# Audit — AVG/compliance · veiligheid & bots · marketing/SEO (juni 2026)

Drie deepdives op de live code (commit `17101f8`). Leidend principe: **minimaal voldoen aan
de regels, gebruiksgemak en toegankelijkheid voorop** — geen frictie toevoegen waar het niet
hoeft. Per bevinding staat het tegengeluid erbij waar "meer compliance/security" de UX zou
schaden. Status: ✅ verwerkt in de branch · ⬜ open / extern.

Genummerde acties staan ook in `feedback-status.md` (#79–#88).

## 1. AVG / cookies (compliance)

**Eindbeeld:** privacy-by-design zit goed in elkaar — geen trackers, geen advertenties,
geen IP-logging voor tracking, account-verwijdering is een volledige cascade. De enige echte
launch-blocker was een onvolledige privacyverklaring; dat is een tekst-fix, geen engineering.

| Bevinding | Ernst | Actie |
|--|--|--|
| **Geen cookiebanner nodig** — alle cookies zijn strikt noodzakelijk (auth-sessie via `@supabase/ssr`). Een banner = pure UX-schade voor nul juridische winst. | n.v.t. | **Bewust NIET doen.** Beantwoord op `/privacy`. |
| Privacyverklaring miste: verantwoordelijke, grondslag, bewaartermijn, volledige verwerkerslijst, kinderbepaling, klachtrecht AP, VS-doorgifte. | 🔴 blocker (brede launch) | ✅ `/privacy` uitgebreid met blok "Details & je rechten (AVG)". |
| Vlaggen werden client-side van **flagcdn.com** gehaald → lekt IP/UA naar een derde. | 🟡 should-fix | ✅ 48 vlaggen self-gehost in `public/assets/flags/`; `flagUrl()` wijst nu lokaal. |
| **Kinderdata** (kid-accounts, AVG art. 8): admin/ouder maakt het account, maar nergens vastgelegd. | 🟡 should-fix | ✅ Gedocumenteerd in `/privacy` (kind = alleen bijnaam/teamnaam/avatar/voortgang, geen e-mail; ouder/beheerder maakt en beheert). |
| Datasoorten in `/privacy` onvolledig (prikbordberichten, avatar, scores ontbraken). | 🟢 nice | ✅ Aangevuld. |
| Bewaartermijn niet benoemd. | 🟢 nice | ✅ Benoemd (zolang account bestaat; opruiming na toernooi/op verzoek). |

**Niet doen (anti-frictie):** consent-checkbox opsplitsen, of een self-service data-export-portaal
bouwen. Handmatig op verzoek via `contact@slimescore.com` volstaat op deze schaal.

**Extern (jij):** bevestig Supabase-projectregio (EU) en de verwerkersovereenkomsten
(Supabase / Vercel / e-mailverzender).

## 2. Veiligheid & bots + technische kwetsbaarheden

**Eindbeeld:** voor zijn schaal degelijk gebouwd. RLS staat aan op elke tabel met zinnige
policies, service-role key is server-only, sync-secret is timing-safe, server-actions checken
auth + eigenaarschap (**geen IDOR**), open-redirect afgevangen, XSS afgedekt door React-escaping
+ DB CHECK-constraints + statische JSON-LD. Geen SQL-injectie/SSRF. De echte gaten zaten in
bot/abuse-weerbaarheid.

| Bevinding | Ernst | Actie |
|--|--|--|
| **Magic-link-aanvraag** heeft geen rate-limit/bot-bescherming → e-mailbombing van derden + massa-fake-accounts via de gebrande afzender. | 🔴 hoog | ⬜ **Extern:** zet Supabase Auth → *Attack Protection* aan (onzichtbare Turnstile/hCaptcha) en stuur de `captchaToken` mee in `signInWithOtp`. Onzichtbaar voor echte gezinnen — **geen interactieve CAPTCHA**. |
| **Kid-codes** waren kort (min. 4) en de code = wachtwoord → raadbaar/brute-forcebaar, geen lockout. | 🔴 hoog | ✅ Minimum naar **8 tekens** (server-validatie + admin-form). Codes als bearer-secret gedocumenteerd. Auto-codes waren al ~40 bit. |
| **Geen Content-Security-Policy.** | 🟡 middel | ✅ CSP **report-only** toegevoegd in `next.config.ts` (met `frame-src` voor de game-subdomeinen). Afdwingen na observatie. |
| Rate-limiter is **fail-open** en alleen op user-id. | 🟡 middel | ⬜ User-keying behouden (gezinsvriendelijk); aanbeveling: log wanneer 'ie fail-open gaat. Het échte abusepad (magic-link) los je op via bot-bescherming, **niet** via per-IP-limieten (die blokkeren gezinnen achter één NAT). |
| `sharp` werd runtime gebruikt maar stond niet in `package.json` (alleen transitief). | 🟢 laag | ✅ Expliciet toegevoegd (`^0.34.5`). |
| Leaderboard-vervuiling door fake-signups. | 🟢 laag | ✅ Geverifieerd: publieke ranglijst filtert via `hasPublicProfile` + lege accounts hebben 0 punten → komen niet in beeld. |

**Sound bevonden (geen actie):** RLS-policies (incl. `kid_accounts`/`admin_audit_log`/`rate_limits`
met RLS aan en géén grants → alleen service-role), secrets-hygiëne, `safeRedirectTarget`,
geparametriseerde queries, banner-upload (auth + owner-check + MIME-gate + 6 MB-cap + re-encode via sharp).

**Niet doen (anti-frictie):** interactieve CAPTCHA, per-IP login-throttling.

## 3. Marketing / vindbaarheid (SEO)

**Eindbeeld:** sterke basis (titel-template, keyword-rijke homepage-H1, robots+sitemap correct,
FAQ-JSON-LD, SSR/ISR-content, `lang="nl"`). De grootste winst zat in een geregresseerde social-card
en ontbrekende metadata op de beste SEO-pagina's.

| Bevinding | Impact | Actie |
|--|--|--|
| Social-preview was een klein vierkant app-icon (`summary`) i.p.v. een brede banner. | 🔴 hoog | ✅ OG-banner **1200×630** + `twitter.card: summary_large_image` hersteld (`og.png` bestond al). |
| `/schema`, `/ranglijst`, `/regels` hadden geen eigen `metadata` → erfden de homepage-titel. | 🔴 hoog | ✅ Eigen title/description **+ self-canonical** toegevoegd; ook `/voorwaarden` en `/games` self-canonical. |
| JSON-LD logo verwees naar twee verschillende paden. | 🟡 middel | ✅ Geünificeerd naar het v2-app-icon. |
| `slimescore.app` zou duplicate content kunnen geven. | 🟡 middel | ⬜ **Extern:** 301-redirect naar `.com` op DNS/Vercel-niveau. |
| Perf: `background-attachment: fixed` + zware ongebruikte PNG's in `/public`. | 🟢 laag | ⬜ Later: asset-hygiëne (controleer of `Hero-bg-square-2400.png` 2,4 MB / `*-1024.png` 2,1 MB live geserveerd worden; zo niet, verwijderen). |
| `/schema` als seizoens-magneet ("wk 2026 speelschema nederlandse tijd"). | 🟡 middel | ✅ Titel/description daarop getuned. |

**Niet doen (anti-frictie):** blog-CMS bouwen, EN-vertaling deze cyclus, nep-`aggregateRating`.
